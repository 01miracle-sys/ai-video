## 1. 代码规范

### 1.1 Python 编码风格

本项目遵循 **PEP 8** 规范，附加约定如下：

**命名规范**：

| 类别 | 规范 | 示例 |
|------|------|------|
| 模块/文件名 | 小写 + 下划线 | `transcribe.py`, `file_utils.py` |
| 类名 | 大驼峰 (PascalCase) | `WhisperTranscriber`, `NotesGenerator` |
| 函数/方法 | 小写 + 下划线 | `generate_notes()`, `format_timestamp()` |
| 变量 | 小写 + 下划线 | `media_path`, `segment_text` |
| 常量 | 全大写 + 下划线 | `DEFAULT_MODEL`, `OLLAMA_API` |
| 私有成员 | 前导下划线 | `_call_ollama()`, `_chunk_size` |

**代码格式**：

```
- 缩进：4 空格（禁止 Tab）
- 行宽：最大 100 字符，优先 88 字符
- 字符串引号：双引号（与 PEP 8 建议一致）
- 类型注解：必须为所有函数参数和返回值添加类型提示
- 导入顺序：标准库 → 第三方库 → 本地模块（每组之间空一行）
```


### 1.2 函数设计原则

1. **单一职责**：一个函数只做一件事。`transcribe.py` 中 `format_timestamp()` 只做时间格式化是好的，后续应将主流程拆分为更小的函数。
2. **避免魔术数字**：使用具名常量代替硬编码数值。
   ```python
   # ❌ 不推荐
   def process(text):
       if len(text) > 4000:
           ...

   # ✅ 推荐
   CHUNK_SIZE = 4000
   def process(text: str) -> str:
       if len(text) > CHUNK_SIZE:
           ...
   ```
3. **提前返回**：尽早处理错误/边界情况，减少嵌套层级。
4. **纯函数优先**：无副作用的函数更容易测试和维护。

### 1.3 错误处理规范

```python
# ✅ 推荐：明确的异常处理
try:
    resp = requests.post(API_URL, json=payload, timeout=30)
    resp.raise_for_status()
    return resp.json().get("response", "")
except requests.Timeout:
    logger.error("Ollama 请求超时")
    raise
except requests.ConnectionError:
    logger.error("无法连接 Ollama 服务")
    raise
```

- 不要吞异常（`except: pass`）
- 给用户可读的错误信息
- 关键失败点使用 `sys.exit(1)` 明确终止

## 2. Git 规范

### 2.1 分支策略

由于是单人项目，采用简化分支策略：

```
main          ← 稳定版本，随时可展示
├── dev       ← 日常开发分支
├── feat/xxx  ← 功能分支（从 dev 切出）
└── fix/xxx   ← 修复分支（从 dev 切出）
```

**工作流程**：

```
1. 从 dev 切出功能分支: feat/add-web-ui
2. 开发完成 → 合并回 dev
3. dev 稳定后 → 合并到 main（发布版本）
```

### 2.2 提交信息规范

采用 **Conventional Commits** 格式：

```
<type>(<scope>): <简短描述>

<详细说明（可选）>
```

**类型（type）**：

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(asr): 添加 VTT 字幕输出格式` |
| `fix` | 修复 bug | `fix(notes): 修复长文本分段时丢失末尾内容` |
| `refactor` | 重构 | `refactor(transcribe): 提取字幕生成函数` |
| `docs` | 文档 | `docs: 添加开发文档和规范` |
| `style` | 代码格式 | `style: 调整缩进和行宽` |
| `test` | 测试 | `test(notes): 添加分段函数单元测试` |
| `chore` | 杂项 | `chore: 添加 requirements.txt` |

**提交示例**：

```
feat(transcribe): 支持多个输出格式 (SRT/VTT/TXT)

- 新增 VTT 格式生成函数
- 通过 --format 参数指定输出格式
- 保持向后兼容，默认输出 SRT + TXT
```

### 2.3 提交粒度

- 每个提交做**一件完整的事**
- 不要混合"修 bug + 加功能"在同一提交
- 频繁提交（完成一个逻辑步骤就提交）

## 3. 文件与目录规范

### 3.1 目录结构演化

当前项目结构较简单，随着功能增加按以下原则重构：

```
原则：
1. src/ 下按功能模块分包（asr/ notes/ subtitle/ utils/）
2. 测试文件与源码分离到 tests/
3. 输出文件统一到 outputs/
4. 配置文件统一到 config/
```

### 3.2 文件命名

```
- Python 文件：小写 + 下划线 (snake_case.py)
- 配置文件：小写 + 下划线 (settings.yaml)
- 文档文件：中文 + .md (开发规范.md)
- 批处理脚本：简短英文 (run.bat)
```

### 3.3 输出文件

所有由工具生成的文件（TXT/SRT/笔记/字幕视频）统一放到 `outputs/` 目录下：

```
outputs/
├── test.txt
├── test.srt
├── test_学习笔记.md
└── test_with_subtitles.mp4
```

## 5. 测试规范

### 5.1 测试原则

- 每个新增功能至少有一个对应的测试
- 修复 bug 时先写复现该 bug 的测试
- 测试应该是**可重复的**（不依赖外部服务的状态）

### 5.2 测试范围

| 类型 | 工具 | 覆盖内容 |
|------|------|----------|
| 单元测试 | pytest | 工具函数（时间格式化、文本分段等） |
| 集成测试 | pytest | ASR → TXT → 笔记全流程 |
| 手动测试 | - | 真实视频转录效果验证 |

### 5.3 测试文件命名

```
tests/
├── test_transcribe.py     # transcribe 模块测试
├── test_generator.py      # 笔记生成测试
├── test_utils.py          # 工具函数测试
└── fixtures/              # 测试用数据
    ├── sample.txt
    └── sample.srt
```

## 6. 项目演进原则

### 6.1 迭代节奏

| 阶段 | 交付物 | 周期 |
|------|--------|------|
| 概念验证 (POC) | 可运行的 CLI 工具 | 已基本完成 |
| MVP | 完整的单视频处理流水线 | 当前阶段 |
| V2 | 字幕烧录 + 配置化 | 下一阶段 |
| V3 | Web 界面 + 批量处理 | 中期目标 |
| V4 | RAG 知识库 + AI 问答 | 远期目标 |

### 6.2 渐进式重构

- 不要一次性大改：每次重构只改一个模块
- 先写测试再重构，确保行为不变
- 保持向后兼容，原有脚本继续可用

### 6.3 "先跑通，再优化"

- 第一阶段：功能正确（不管效率）
- 第二阶段：代码可读（重构、加注释）
- 第三阶段：性能优化（缓存、并发、模型调优）
- 第四阶段：体验打磨（Web UI、进度显示）
